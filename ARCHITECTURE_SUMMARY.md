# Архитектура совместного редактирования (Tiptap + Yjs + Hocuspocus)

В этом документе зафиксированы ключевые архитектурные решения по внедрению совместного редактирования, работе с вебсокетами и разделению зон ответственности в рамках микросервисного подхода (DDD).

## 1. Единый шлюз WebSockets (`ws-gateway`)
Вместо того чтобы плодить множество микросервисов для вебсокетов, мы используем **единый `ws-gateway`**, который выполняет маршрутизацию соединений на основе HTTP-путей.

- **Адаптер:** Используется `WsAdapter` из `@nestjs/platform-ws` вместо стандартного Socket.IO. Это обязательно, так как Hocuspocus требует работы по чистым (native) вебсокетам.
- **Маршрутизация (Порты и Адаптеры):**
  - `ws://.../collaboration` — обрабатывается классом `HocuspocusGateway` (направляет сырые сокеты в инстанс Hocuspocus Server).
  - `ws://.../notifications` — обрабатывается классом `NotificationsGateway` (передает глобальные JSON-уведомления).
- **Почему не одно соединение?** Протокол Yjs — это сложный бинарный поток, привязанный к жизненному циклу конкретного документа. Глобальные уведомления (статус юзера, чат) — это JSON, привязанный к жизненному циклу вкладки приложения. Их физическое разделение избавляет от необходимости писать сложный мультиплексор и позволяет Hocuspocus работать в "коробочном" режиме.

## 2. Единый Агрегат Заметки (`note-service`)
Согласно правилам предметно-ориентированного проектирования (DDD), Заметка (ее метаданные и контент) является **единым Агрегатом**. Разделение ее на разные микросервисы исключительно из-за технических различий в нагрузке является антипаттерном, так как разрывает транзакционную границу.

- **Ответственность:** `note-service` является единственным владельцем (Ограниченным Контекстом) сущности Заметки. Он управляет как метаданными (название, права доступа), так и контентом (бинарный слепок CRDT).
- **Хранение:** Микросервис самостоятельно решает техническую задачу хранения: например, метаданные хранятся в реляционных таблицах PostgreSQL, а бинарники Yjs (`bytea`) — в соседней колонке или специализированной таблице в той же БД.
- **Транзакционная целостность:** При удалении заметки в `note-service` ее бинарные данные гарантированно удаляются в рамках одной транзакции (избегая сложных Саг и распределенных транзакций).
- `ws-gateway` выступает лишь транспортным слоем, который обращается к `note-service` по gRPC или RMQ для получения и сохранения бинарных слепков.

## 3. Оркестрация через Frontend
Связывание слоев воедино происходит на стороне клиента:
1. Фронтенд создает/запрашивает заметку через `note-service` по REST и получает уникальный UUID (например, `123e456...`).
2. Фронтенд использует этот же UUID для инициализации `HocuspocusProvider`:
   ```ts
   const provider = new HocuspocusProvider({
     url: 'ws://api.../collaboration',
     name: noteId, // UUID заметки выступает идентификатором
   });
   ```
3. `ws-gateway` авторизует соединение и по этому UUID запрашивает бинарный слепок у `note-service`.

## 4. Полнотекстовый поиск и AI-векторизация (CQRS и Eventual Consistency)
Поскольку Yjs хранит данные в бинарном формате, ИИ и классический поиск (ElasticSearch, pgvector) не могут напрямую их индексировать.
Решение проблемы происходит с использованием паттерна **CQRS**:

- **Write Model:** Быстрый бинарный протокол Yjs, который фиксирует состояние в `note-service`.
- **Read Model (Проекции):** Внутри `ws-gateway` (Hocuspocus) раз в N секунд срабатывает хук `onStoreDocument`. Он достает из оперативной памяти готовый инстанс `Y.Doc` и извлекает из него **чистый текст** (`plainText`).
- **События предметной области (Domain Events):** Генерируется RMQ событие `note.content_updated`, которое отправляется в `topic exchange`.
- **Подписчики:** Микросервисы поиска (`search-service`) и искусственного интеллекта (`ai-service`) слушают это событие и асинхронно обновляют свои поисковые индексы и векторные базы (pgvector).

Это обеспечивает превосходную масштабируемость и итоговую согласованность (Eventual Consistency) без замедления работы самого редактора.
